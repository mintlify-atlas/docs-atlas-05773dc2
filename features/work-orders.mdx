# Work Order Management

Work Orders (Órdenes de Trabajo or OT) are the operational core of DRAIT Mini-MES. They represent production jobs from initial planning through delivery, tracking all resources, time, materials, and operational events.

## Overview

Work orders flow through multiple states representing their lifecycle in the shop floor:

- **PENDIENTE** — Newly created, awaiting planning
- **PLANIFICADA** — Scheduled with dates and resources
- **EN_PROCESO** — Active production underway
- **PAUSADA** — Temporarily halted
- **FINALIZADA** — Production complete, pending delivery
- **ENTREGADA** — Delivered to customer with signed delivery note
- **CANCELADA** — Cancelled before completion
- **RETRABAJO** — Requires rework due to non-conformance

## Creating Work Orders

### From Scratch

<Steps>
  <Step title="Open the Work Orders page">
    Navigate to **Production** → **Work Orders** in the main navigation.
  </Step>
  
  <Step title="Click 'Nueva Orden'">
    Open the creation modal by clicking the "+ Nueva Orden" button.
  </Step>
  
  <Step title="Fill required fields">
    - **Código**: Internal code (e.g., `OT-2026-045`)
    - **Cliente**: Select from existing clients
    - **Título**: Short descriptive title
    - **Descripción**: Detailed work description
    - **Tiempo estimado**: Estimated time in minutes
    - **Costo estimado**: Estimated total cost
    - **Fecha compromiso**: Optional commitment date
  </Step>
  
  <Step title="Submit">
    Click "Crear OT" to generate the work order.
  </Step>
</Steps>

### From Approved Quotation

Work orders can be automatically generated from approved quotations:

<Steps>
  <Step title="Open quotation management">
    In the Quotations page, click "⚙️ Gestionar" on an approved quotation.
  </Step>
  
  <Step title="Convert to work order">
    In the "Convertir a Orden de Trabajo" section:
    - Review the suggested OT code
    - Optionally set a commitment date
    - Click "⚡ Confirmar y Enviar a Producción"
  </Step>
  
  <Step title="Work order created">
    The system creates a work order with:
    - All quotation details copied
    - Status set to PENDIENTE
    - Link maintained to source quotation
  </Step>
</Steps>

<Info>
  Converting from quotations ensures cost estimates and technical details carry forward into production planning.
</Info>

## Managing Work Orders

### Assignment to Resources

Work orders must be assigned to human resources (operators) to begin work:

<CardGroup cols={2}>
  <Card title="Assign Operators" icon="user-plus">
    1. Click "⚙️ Gest." on a work order
    2. Select operator from "Asignar Puesto" dropdown
    3. Click "Asignar"
    
    The system creates a `WorkOrderAssignment` linking the operator's user account and corresponding resource.
  </Card>
  
  <Card title="View Assignments" icon="users">
    Active assignments appear in:
    - Work order detail panel (right sidebar)
    - Operator's personal dashboard
    - Resource utilization reports
  </Card>
</CardGroup>

### Status Transitions

Work order status changes are tracked in `OperationLog` with event types:

- **OT_CREADA** — Work order created
- **ASIGNADO** — Resource/operator assigned  
- **INICIO** — Work started
- **PAUSA** — Work paused (with optional reason)
- **REANUDACION** — Work resumed
- **FINALIZACION** — Work completed
- **CAMBIO_ESTADO** — Manual status change

### Material Consumption

As production progresses, operators register material consumption:

<Steps>
  <Step title="Navigate to operator dashboard">
    Operators access their assigned work orders at `/operator-dashboard`.
  </Step>
  
  <Step title="Log material usage">
    For each material consumed:
    - Select material from inventory
    - Enter quantity used
    - Add optional notes
    - Submit consumption record
  </Step>
  
  <Step title="Automatic cost tracking">
    The system creates a `MaterialConsumption` record with:
    - `unitCostSnapshot`: Material cost frozen at consumption time
    - Link to work order for cost aggregation
    - Timestamp for traceability
    - Automatic inventory deduction
  </Step>
</Steps>

<Warning>
  Material costs are snapshotted at consumption time to preserve accurate historical costing even if material prices change later.
</Warning>

## Delivery and Closure

### Completing Work Orders

When production is finished, supervisors complete the delivery process:

<Steps>
  <Step title="Open work order management">
    Click "⚙️ Gest." on a FINALIZADA work order.
  </Step>
  
  <Step title="Complete delivery checklist">
    In the "Completar (Remito)" section:
    - Fill delivery checklist items
    - Set delivery date/time
    - Check "Remito Firmado" if customer signed
    - Optionally upload signed delivery note (PDF/image)
  </Step>
  
  <Step title="Emit delivery">
    Click "Emitir" to:
    - Change status to ENTREGADA
    - Record delivery timestamp
    - Store delivery note attachment
    - Close work order
  </Step>
</Steps>

### Delivery Attachments

Work orders support file attachments for delivery documentation:

<Tabs>
  <Tab title="Upload">
    ```typescript
    // Endpoint: POST /api/work-orders/:id/attachment
    // Content-Type: multipart/form-data
    
    const formData = new FormData();
    formData.append('file', selectedFile);
    await api.post(`/work-orders/${workOrderId}/attachment`, formData);
    ```
    
    Supported formats: PDF, images (PNG, JPG). Max size: 10MB.
  </Tab>
  
  <Tab title="View">
    Uploaded files are accessible at:
    ```
    /api/work-orders/:id/attachment/view
    ```
    
    The `attachmentUrl` field stores the relative path.
  </Tab>
  
  <Tab title="Delete">
    ```typescript
    // Endpoint: DELETE /api/work-orders/:id/attachment
    await api.delete(`/work-orders/${workOrderId}/attachment`);
    ```
  </Tab>
</Tabs>

## Work Order Details

### Core Fields

| Field | Type | Description |
|-------|------|-------------|
| `code` | String | Unique visible code (e.g., OT-2026-001) |
| `title` | String | Work order title |
| `description` | String | Detailed work description |
| `status` | Enum | Current operational status |
| `priority` | Integer | Priority level (1=high, 5=low) |
| `estimatedTimeMin` | Integer | Estimated time in minutes |
| `estimatedCost` | Decimal | Estimated production cost |
| `commitmentDate` | DateTime | Promised delivery date |
| `plannedDate` | DateTime | Internal planning date |
| `startedAt` | DateTime | Actual start timestamp |
| `finishedAt` | DateTime | Actual completion timestamp |
| `deliveredAt` | DateTime | Delivery timestamp |
| `deliveryChecklist` | String | Delivery verification checklist |
| `isSigned` | Boolean | Whether delivery note is signed |
| `attachmentUrl` | String | Path to delivery document |

## Deletion Rules

Work orders can only be deleted under strict conditions:

- Status must be PENDIENTE or CANCELADA
- No resource assignments exist
- No material consumption records
- No operation logs recorded

<Warning>
  Deletion is permanent and cannot be undone. Use CANCELADA status instead to preserve historical records.
</Warning>

## Best Practices

<Card title="Workflow Recommendations" icon="lightbulb">
  1. **Always set commitment dates** for customer expectations
  2. **Assign resources early** to enable operator planning
  3. **Log material consumption** as work progresses, not after
  4. **Use pause reasons** to track downtime causes
  5. **Attach signed delivery notes** for legal documentation
  6. **Review alerts** in dashboard for delayed orders
  7. **Use quotation conversion** to maintain estimate accuracy
</Card>

## API Endpoints

<Tabs>
  <Tab title="List & Retrieve">
    ```typescript
    // List all work orders
    GET /api/work-orders
    
    // Get single work order with full details
    GET /api/work-orders/:id
    ```
  </Tab>
  
  <Tab title="Create & Update">
    ```typescript
    // Create work order
    POST /api/work-orders
    Body: { code, title, description, clientId, estimatedTimeMin, estimatedCost, ... }
    
    // Create from quotation
    POST /api/work-orders/from-quotation/:quotationId
    Body: { code, commitmentDate? }
    
    // Update work order
    PATCH /api/work-orders/:id
    Body: { title?, description?, commitmentDate?, ... }
    
    // Change status
    PATCH /api/work-orders/:id/status
    Body: { status: WorkOrderStatus }
    ```
  </Tab>
  
  <Tab title="Operations">
    ```typescript
    // Assign resource
    POST /api/work-orders/:id/assignments
    Body: { resourceId, userId? }
    
    // Log operational event
    POST /api/work-orders/:id/events
    Body: { eventType, eventAt, pauseReason?, note? }
    
    // Register material consumption
    POST /api/work-orders/:id/consumptions
    Body: { materialId, quantity, note? }
    
    // Close delivery
    POST /api/work-orders/:id/close-delivery
    Body: { deliveredAt?, deliveryChecklist?, isSigned? }
    ```
  </Tab>
  
  <Tab title="Attachments">
    ```typescript
    // Upload delivery attachment
    POST /api/work-orders/:id/attachment
    Content-Type: multipart/form-data
    
    // Delete attachment
    DELETE /api/work-orders/:id/attachment
    ```
  </Tab>
  
  <Tab title="Delete">
    ```typescript
    // Delete work order (restricted)
    DELETE /api/work-orders/:id
    ```
  </Tab>
</Tabs>
