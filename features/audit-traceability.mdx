# Audit Logs and Traceability

DRAIT Mini-MES maintains comprehensive audit trails for all critical system operations. Every create, update, delete, and status change is logged with user attribution, timestamps, and before/after state capture.

## Overview

The audit system provides:

- **Complete history**: Every change recorded with context
- **User attribution**: Who made each change
- **State capture**: Before and after values for changes
- **Compliance**: Supports regulatory and quality management requirements
- **Forensics**: Investigate issues and understand system state evolution

<Info>
  Audit logs are append-only. Records cannot be edited or deleted, ensuring immutable history.
</Info>

## Action Types

Audit logs capture these action types:

| Action | Description | Example |
|--------|-------------|----------|
| **CREATE** | New entity created | Creating a new work order |
| **UPDATE** | Entity modified | Updating client contact info |
| **DELETE** | Entity deleted | Deleting a quotation |
| **STATUS_CHANGE** | Status field changed | Work order: PENDIENTE → EN_PROCESO |
| **ASSIGN** | Resource assigned | Operator assigned to work order |
| **DELIVERY_CLOSE** | Work order delivered | Completing delivery with signed remito |
| **PASSWORD_CHANGE** | User password reset | Admin resetting operator password |
| **ROLE_CHANGE** | User role modified | Promoting OPERARIO to SUPERVISOR |
| **SETTINGS_CHANGE** | Company settings updated | Changing deviation thresholds |
| **LOGIN** | Authentication event | User login to system |

## Audit Log Structure

```prisma
model AuditLog {
  id         String
  companyId  String
  userId     String?         // User who performed action (null for system)
  entityType AuditEntityType // What was changed
  entityId   String          // ID of changed entity
  action     AuditActionType // What action occurred
  before     Json?           // State before change
  after      Json?           // State after change
  metadata   Json?           // Additional context
  createdAt  DateTime
  
  company    Company
  user       User?
}
```

### State Capture

For UPDATE, DELETE, and STATUS_CHANGE actions, the system captures:

<Tabs>
  <Tab title="Before State">
    The `before` field contains the entity state prior to the change:
    
    ```json
    {
      "status": "PENDIENTE",
      "priority": 3,
      "commitmentDate": "2026-03-15T00:00:00.000Z"
    }
    ```
    
    Only fields being changed are included.
  </Tab>
  
  <Tab title="After State">
    The `after` field contains the new state:
    
    ```json
    {
      "status": "EN_PROCESO",
      "priority": 3,
      "commitmentDate": "2026-03-15T00:00:00.000Z"
    }
    ```
    
    Enables precise diff calculation.
  </Tab>
  
  <Tab title="Metadata">
    The `metadata` field stores action-specific context:
    
    ```json
    {
      "ipAddress": "192.168.1.100",
      "userAgent": "Mozilla/5.0...",
      "reason": "Customer requested priority increase"
    }
    ```
    
    Extensible for future enhancements.
  </Tab>
</Tabs>

<Warning>
  Sensitive data like passwords is NEVER stored in audit logs, even in hashed form. Password changes log the action only.
</Warning>

### Investigating User Changes

To audit user account modifications:

1. Filter by **Entity Type**: USER
2. Review timeline of actions:
   - CREATE: User account created
   - ROLE_CHANGE: Role elevated or demoted
   - PASSWORD_CHANGE: Password reset by admin
   - UPDATE: Profile information changed

<Tabs>
  <Tab title="Role Change Example">
    ```json
    {
      "entityType": "USER",
      "entityId": "clx1234...",
      "action": "ROLE_CHANGE",
      "userId": "clx9876...",  // Admin who made change
      "before": { "role": "OPERARIO" },
      "after": { "role": "SUPERVISOR" },
      "createdAt": "2026-03-10T09:15:00Z"
    }
    ```
    
    Shows operator promoted to supervisor by admin user.
  </Tab>
  
  <Tab title="Password Reset">
    ```json
    {
      "entityType": "USER",
      "entityId": "clx1234...",
      "action": "PASSWORD_CHANGE",
      "userId": "clx9876...",  // Admin who reset password
      "before": null,
      "after": null,  // No password data stored
      "metadata": {
        "reason": "User forgot password"
      },
      "createdAt": "2026-03-12T11:20:00Z"
    }
    ```
    
    Password content is never logged for security.
  </Tab>
</Tabs>

## Best Practices

<Card title="Audit Log Usage" icon="lightbulb">
  **For Administrators:**
  - Review audit logs weekly for unusual patterns
  - Monitor DELETE and ROLE_CHANGE actions closely
  - Investigate multiple failed LOGIN attempts
  - Use audit logs for training on common errors
  - Export logs periodically for external archival
  
  **For Quality Managers:**
  - Reference audit logs during non-conformance investigations
  - Use work order history to identify process improvements
  - Document corrective actions with audit trail evidence
  - Include relevant audit logs in quality reports
  
  **For Compliance:**
  - Establish retention policies (recommend 7 years minimum)
  - Implement automated audit log backups
  - Define access controls for sensitive audit data
  - Create standard audit report templates
  - Train staff on audit log importance
  
  **Security:**
  - Restrict audit log deletion permissions (should be none)
  - Monitor LOGIN events for unauthorized access attempts
  - Alert on after-hours administrative actions
  - Use audit logs as input to security incident response
</Card>

## Performance Considerations

<Warning>
  Audit logs grow continuously. Implement archival strategy for production deployments.
</Warning>

**Recommendations:**

1. **Indexing**: Database indexes on:
   - `companyId + createdAt` (time-based queries)
   - `entityType + entityId` (entity history)
   - `userId + createdAt` (user activity)
   - `action + createdAt` (action-based reports)

2. **Archival**: Move logs older than 1 year to cold storage:
   - Compress to JSON files
   - Store in S3 or similar
   - Maintain searchability via data lake tools

3. **Pagination**: Always use `limit` parameter:
   - Default: 100 records
   - Maximum: 1000 records per request
   - Use cursor-based pagination for large exports

4. **Caching**: Recent logs (last 24h) can be cached:
   - Redis cache for audit log dashboard
   - Invalidate on new audit log creation

```typescript
GET /api/audit-logs
Query params:
  - entityType?: AuditEntityType
  - entityId?: string
  - action?: AuditActionType
  - userId?: string
  - limit?: number (default 100, max 1000)
  - offset?: number

Response: AuditLog[]

// Get single audit log
GET /api/audit-logs/:id

Response: AuditLog

// Note: No POST, PATCH, or DELETE endpoints
// Audit logs are created automatically by system

```

### Creating Audit Logs (Backend)

Audit logs are created automatically by the backend audit service:

```typescript
// Example: Creating audit log in backend
import { AuditService } from '@/modules/audit/audit.service';

class WorkOrderService {
  constructor(private auditService: AuditService) {}
  
  async updateWorkOrder(id: string, data: UpdateDto, userId: string) {
    const before = await this.workOrderRepo.findById(id);
    const updated = await this.workOrderRepo.update(id, data);
    
    // Automatically create audit log
    await this.auditService.log({
      entityType: 'WORK_ORDER',
      entityId: id,
      action: 'UPDATE',
      userId,
      before: { status: before.status, priority: before.priority },
      after: { status: updated.status, priority: updated.priority }
    });
    
    return updated;
  }
}
```

<Info>
  Frontend developers don't directly create audit logs. They're generated automatically by backend services.
</Info>
