## Overview

DRAIT Mini-MES uses PostgreSQL 16 as its database and Prisma as the ORM. This guide covers database initialization, migrations, and seeding.

## Prisma Schema

The database schema is defined in `apps/backend/prisma/schema.prisma`:

```prisma schema.prisma
generator client {
  provider = "prisma-client-js"
}

datasource db {
  provider = "postgresql"
  url      = env("DATABASE_URL")
}
```

### Key Enums

```prisma
enum UserRole {
  DUENO      // Owner with full access
  SUPERVISOR // Supervises operations
  OPERARIO   // Shop floor operator
  ADMIN      // Technical administrator
}

enum WorkOrderStatus {
  PENDIENTE     // Pending
  PLANIFICADA   // Planned
  EN_PROCESO    // In progress
  PAUSADA       // Paused
  FINALIZADA    // Finished
  ENTREGADA     // Delivered
  CANCELADA     // Cancelled
  RETRABAJO     // Rework
}

enum ResourceType {
  HUMANO   // Human resource
  MAQUINA  // Machine/equipment
}

enum OperationEventType {
  OT_CREADA         // Work order created
  ASIGNADO          // Resource assigned
  INICIO            // Start
  PAUSA             // Pause
  REANUDACION       // Resume
  FINALIZACION      // Completion
  CAMBIO_ESTADO     // Status change
  MATERIAL_CONSUMIDO // Material consumed
  NOTA              // Note
}
```

### Core Models

| Model | Description |
|-------|-------------|
| `Company` | Tenant organization. All data is isolated by `companyId` |
| `User` | System users with roles (OPERARIO, SUPERVISOR, DUENO, ADMIN) |
| `Client` | Customer organizations |
| `Quotation` | Sales quotations that generate work orders |
| `WorkOrder` | Core entity - manufacturing orders |
| `Resource` | Human and machine resources |
| `Material` | Inventory items with stock tracking |
| `WorkOrderEvent` | Timestamped events (START, PAUSE, RESUME, FINISH) |
| `MaterialConsumption` | Material usage per work order |
| `OperationLog` | Operational audit trail |
| `AuditLog` | System-wide audit log with before/after snapshots |

## Initial Setup

<Steps>
  <Step title="Start Database Container">
    The database container must be running before migrations:

    ```bash
    docker compose up -d drait-db
    ```

    Wait for the health check to pass:
    ```bash
    docker compose ps drait-db
    ```

    Status should show `healthy`.
  </Step>

  <Step title="Generate Prisma Client">
    Generate the TypeScript client from the schema:

    ```bash
    docker compose exec backend npm run prisma:generate
    ```

    This creates type-safe database access code in `node_modules/@prisma/client`.
  </Step>

  <Step title="Run Migrations">
    Apply the database schema:

    ```bash
    docker compose exec backend npm run prisma:migrate
    ```

    <Info>
      This command runs two operations:
      1. Normalizes enum names (see Enum Normalization below)
      2. Pushes schema to database with `prisma db push`
    </Info>
  </Step>

  <Step title="Verify Schema">
    Check that tables were created:

    ```bash
    docker compose exec drait-db psql -U drait -d drait_mes -c "\dt"
    ```

    You should see tables like `Company`, `User`, `WorkOrder`, etc.
  </Step>
</Steps>

## Migration Commands

The `package.json` defines Prisma scripts:

```json apps/backend/package.json
{
  "scripts": {
    "prisma:generate": "prisma generate",
    "prisma:normalize-enums": "prisma db execute --file prisma/enum-rename.sql --schema prisma/schema.prisma",
    "prisma:migrate": "npm run prisma:normalize-enums && prisma db push",
    "prisma:seed": "node dist/prisma/seed.js"
  },
  "prisma": {
    "seed": "node dist/prisma/seed.js"
  }
}
```

### Generate Client

```bash
docker compose exec backend npm run prisma:generate
```

Generates TypeScript types and client code. Run this after:
- Cloning the repository
- Changing the schema
- Installing dependencies

### Normalize Enums

```bash
docker compose exec backend npm run prisma:normalize-enums
```

Executes `prisma/enum-rename.sql` to normalize PostgreSQL enum names. This ensures enum values match the Prisma schema exactly.

<Note>
  This step is automatically included in `prisma:migrate`.
</Note>

### Push Schema Changes

```bash
docker compose exec backend npm run prisma:migrate
```

Pushes the schema to the database without creating migration files. This:
1. Normalizes enums
2. Creates/updates tables
3. Applies schema changes

<Warning>
  `prisma db push` is used instead of `prisma migrate dev` because the project uses custom enum normalization. In production, this approach is suitable for small teams with direct database access.
</Warning>

### Seed Database

```bash
docker compose exec backend npm run prisma:seed
```

Loads demo data from `apps/backend/prisma/seed.ts`. See [Seeding Data](#seeding-data) below.

## Database Backup

### Create Backup

```bash
# Backup to file
docker compose exec drait-db pg_dump -U drait -d drait_mes > backup-$(date +%Y%m%d-%H%M%S).sql

# Backup with compression
docker compose exec drait-db pg_dump -U drait -d drait_mes | gzip > backup-$(date +%Y%m%d-%H%M%S).sql.gz
```

### Restore Backup

```bash
# From SQL file
cat backup-20260313-120000.sql | docker compose exec -T drait-db psql -U drait -d drait_mes

# From compressed file
gunzip -c backup-20260313-120000.sql.gz | docker compose exec -T drait-db psql -U drait -d drait_mes
```

<Warning>
  Stop the backend before restoring to prevent connection errors:
  ```bash
  docker compose stop backend
  # restore database
  docker compose start backend
  ```
</Warning>

### Automated Backup Script

```bash backup.sh
#!/bin/bash
BACKUP_DIR="/path/to/backups"
DATE=$(date +%Y%m%d-%H%M%S)
FILE="$BACKUP_DIR/drait-mes-$DATE.sql.gz"

mkdir -p $BACKUP_DIR
docker compose exec -T drait-db pg_dump -U drait -d drait_mes | gzip > $FILE

echo "Backup saved: $FILE"

# Keep only last 7 days
find $BACKUP_DIR -name "drait-mes-*.sql.gz" -mtime +7 -delete
```

Add to crontab for daily backups:
```bash
0 2 * * * /path/to/backup.sh
```

## Database Maintenance

### Access Database Shell

```bash
docker compose exec drait-db psql -U drait -d drait_mes
```

### Common Queries

```sql
-- List all tables
\dt

-- Describe a table
\d "WorkOrder"

-- Count work orders by status
SELECT status, COUNT(*) FROM "WorkOrder" GROUP BY status;

-- List all companies
SELECT id, name, "legalName" FROM "Company";

-- Check database size
SELECT pg_size_pretty(pg_database_size('drait_mes'));
```

### Reset Database

<Warning>
  This deletes ALL data permanently!
</Warning>

```bash
# Stop backend
docker compose stop backend

# Drop and recreate database
docker compose exec drait-db psql -U drait -c "DROP DATABASE drait_mes;"
docker compose exec drait-db psql -U drait -c "CREATE DATABASE drait_mes;"

# Run migrations
docker compose exec backend npm run prisma:migrate

# Seed data
docker compose exec backend npm run prisma:seed

# Start backend
docker compose start backend
```

## Troubleshooting

### Migration Fails

**Error:** `Can't reach database server`

**Solution:** Verify database is running and `DATABASE_URL` is correct:
```bash
docker compose ps drait-db
echo $DATABASE_URL
```

### Enum Type Errors

**Error:** `Invalid enum value`

**Solution:** Run enum normalization:
```bash
docker compose exec backend npm run prisma:normalize-enums
```

### Seed Script Fails

**Error:** `Cannot find module 'dist/prisma/seed.js'`

**Solution:** Build the backend first:
```bash
docker compose exec backend npm run build
docker compose exec backend npm run prisma:seed
```

### Connection Pool Exhausted

**Error:** `Error: Connection pool timeout`

**Solution:** Restart backend to reset connection pool:
```bash
docker compose restart backend
```
